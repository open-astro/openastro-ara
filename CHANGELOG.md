# Changelog

All notable changes to OpenAstro Ara are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).
Versioning follows [Semantic Versioning](https://semver.org/spec/v2.0.0.html) —
pre-1.0, breaking changes can happen in any release per the playbook §0.6.

For Ara-specific release tagging convention (`v0.0.1-ara.N`), see
[design/PORT_PLAYBOOK.md §34.5](design/PORT_PLAYBOOK.md). Phase boundary tags
(`phase-N-complete`) are NOT release tags — they're internal milestone markers
used by the port's `port/ara → master` promotion cadence (playbook §22.0).

## How to update this file

Every PR that lands a user-visible change adds a bullet to the **[Unreleased]**
section as part of its diff (convention enforced by the port-driver skill, see
`.claude/skills/port-driver/SKILL.md` step 4). Pick the matching subsection:

- **Added** for new features, endpoints, settings, or capabilities.
- **Changed** for changes in existing behavior.
- **Deprecated** for soon-to-be removed features.
- **Removed** for now-removed features or files.
- **Fixed** for bug fixes.
- **Security** for vulnerability fixes.

Mechanical PRs (pure renames, pure deletes during demolition, doc-only edits to
internal design files) don't need a CHANGELOG entry. When in doubt, add one.

At release time (e.g. `v0.0.1-ara.1`), the **[Unreleased]** section is renamed
to `[0.0.1-ara.1] - YYYY-MM-DD` and a fresh `[Unreleased]` placeholder is added
at the top. This happens in the same commit that pushes the release tag.

## [Unreleased]

### Added
- **§18.I — center on any sky position over REST.** `POST /api/v1/platesolve/center` with an RA/Dec slews the mount there and iterates capture → plate solve → sync → re-slew until it's actually on target — the same loop the meridian-flip and safety re-centers use, now callable directly. Centering takes minutes, so the call returns immediately with a background job: poll `GET /api/v1/jobs/{id}` for per-solve-attempt progress, cancel with DELETE, and a run that can't converge (or a mis-configured solver, or a disconnected mount) lands as a failed job with the reason spelled out. One centering runs at a time — a second request joins the one in flight.
- **§35 — after a weather pause clears, the mount points at your target again before the sequence resumes.** The auto-resume used to release the run wherever the park had left the mount and tell you to "verify pointing." Now, when conditions stay safe through the resume delay, the daemon unparks, re-centers on the paused run's target by plate solve (the target the pause actually interrupted, in a multi-target plan), and only then releases the run — so the first post-resume frames aren't wasted on empty sky. Best-effort and honest: no solver or no coordinate target means the old verify-pointing warning; a re-center that doesn't converge resumes anyway and says so; a relapse to unsafe mid-re-center re-parks and keeps the run paused for the next safe window; and a 10-minute bound guarantees a slow rig can never hold the resume hostage.
- **§57 — a watchdog-killed flip slew is now reported honestly, and every mount stop is timed.** When the meridian-flip watchdog aborts a stalled flip slew, the event stream now says `slew_aborted` with `reason: "watchdog"` instead of pretending the slew completed — so WILMA's overlay and any log reader see what actually happened. And every abort (panic button or watchdog) logs a structured line with the server-side issue latency, so the §57.5 panic-to-stop budget is tunable from real incident data.
- **§68 — an empty equipment scan now tells you what's actually wrong.** When device discovery comes back empty after setup, the most common cause isn't a missing driver — it's AlpacaBridge (the equipment hub) not installed or its service stopped. The device chooser's empty state now says exactly that and shows the fix (`sudo apt install alpaca-bridge`, `systemctl status alpaca-bridge`) plus the subnet note for drivers on another host, instead of a generic "no devices found" shrug.
- **§57 — the Stop Mount panic button.** Whenever the mount is slewing under server control — a sequence's go-to, a meridian flip, crash recovery, park — a prominent red STOP MOUNT button surfaces over whatever tab you're on (Space on desktop works too), showing where the mount is headed. One tap, no confirmation gate: the mount halts in place and any running sequence pauses at its next instruction boundary, while cooler, guider, and everything else keep running — this is "this slew is going somewhere I don't like", not the Emergency Stop's "kill everything". A modal then shows where the mount actually stopped and offers Resume / Skip this target / End session against the paused run. The button appears only during slews, so it never crowds the UI, and it clears itself if the server link drops.
- **§57 — slews are now announced, and the panic stop pauses the night properly.** The daemon publishes slew lifecycle events (`telescope.slew_started` with the commanded target, `telescope.slew_complete` with the duration, `telescope.slew_aborted` the instant a stop lands) observed from the mount's own motion state — so every slew source is covered: a sequence's go-to, a meridian flip, crash recovery, park, or a raw API call. And the mount-stop endpoint now does what the panic button promises beyond halting the mount: any running sequence is paused at the next instruction boundary instead of blindly continuing into exposures pointed at the wrong sky. This is the server half of the Stop Mount button; the in-app overlay button builds on these events next.
- **§31 — the time-sync waterfall now has a face, and a last resort.** Settings → Safety → Site grows a Time sync section: whether the observatory clock is synced (and from what, at what trust), the tracked clock offset, the position the daemon knows, and whether a USB GPS is detected — with Retry, a one-click "push device time", and a manual-entry modal (UTC date/time plus optional lat/long/altitude) for the truly offline field setup: no internet on anything, no GPS dongle, just you and a wristwatch. Manual entries apply at low trust, so schedule-based instructions will still warn before trusting them. When the server is unsynced, the section spells out the §31 options (plug a USB GPS into the Pi / push this device's clock / enter manually). Findable from ⌘K via "time sync", "gps", "utc", or "clock".
- **§31 — a $20 USB GPS dongle now keeps the observatory clock (and site location) right, hands-free.** Plug one into the Pi and the daemon finds it on its own: it watches the serial ports a GPS presents as, reads the satellite fix, and applies a high-trust time sync — plus your actual latitude/longitude into the active profile — with no app connected at all. It checks every couple of minutes until it has a sync and re-syncs hourly after that, so multi-day unattended projects never drift. Garbled receivers, busy ports, and cold units with nonsense dates are all quietly skipped.
- **§31 — WILMA now sets the observatory's clock automatically.** On every connect (and reconnect), the app asks the daemon whether it holds a fresh, trustworthy time sync and quietly pushes this device's clock when it doesn't — a phone or laptop on Wi-Fi is NTP-synced, which covers a Pi without internet or a real-time clock for the vast majority of sessions. Silent and best-effort: nothing to configure, no prompt, and a failed push just retries on the next connect.
- **§31 — the daemon can now be told what time it is.** A Pi without internet or an RTC drifts; the new time-sync surface lets WILMA (or anything else) push UTC time and location: `GET /api/v1/server/time-sync` reports whether the server holds a fresh, trustworthy sync (plus any tracked clock offset), and `POST` applies one — setting the OS clock when the server binary has the CAP_SYS_TIME capability (see DEPLOY.md), or honestly tracking the offset when it doesn't. A pushed location lands straight in the active profile's site settings, so twilight times and Tonight's Sky pick it up immediately. Trust levels follow the §31 ladder (a manual entry can never claim more than low trust). The USB-GPS self-sync and the in-app sync prompts build on this next.
- **§61.10 — the ⌘K palette now runs commands.** Five actions join the navigation hits: "Open Image Library", "Open Calibration", "Open Stats", "Back up & restore", and "Run profile wizard" — every top-bar launcher, now reachable from the keyboard. Action rows carry a bolt icon.
- **§61.10 — the ⌘K palette can now navigate.** Typing "go to run", "planning", "live view", or "preferences" (and similar intent words) surfaces a navigation hit that jumps straight to that main tab — the first slice of growing the settings search into a full command palette. Navigation rows carry an arrow icon so they read apart from settings and help hits.
- **§43 — creating a backup no longer ties up the app for the whole packaging run.** Since backups began carrying the frames-catalog snapshot they can take real time on a long-lived library, and the create request used to hold the connection open the entire while (with a 2-minute client timeout as the only guard). Create now returns immediately and packages in the background — the button's spinner tracks the daemon's actual progress, a packaging failure shows the daemon's own error message instead of a generic timeout, and other daemons/clients can watch the new `backup.create.*` events or poll `GET /api/v1/backup/create-status`. Also new: a disk-space pre-flight refuses a backup up front (HTTP 507) when the disk clearly can't hold it, and retrying a timed-out create request with the same idempotency key re-joins the running backup instead of starting a second one.
- **§30.7.4 — the profile now remembers when guider calibration files were last built.** A completed dark-library or defect-map build stamps a `calibration_state` record in the active profile (valid + built-at timestamp per file), so "do I have usable darks, and how old are they?" survives daemon restarts instead of requiring a live guider query. Readable at `GET /api/v1/profile/calibration-state`; the stamps follow the profile across profile switches (a profile that never built darks reads as not-built), and profile-share export strips them — the donor's dark files live on the donor's guider daemon, so "valid" would mislead the recipient. A failed build never stamps.
- **§63.4 — deleting an ARA profile now cleans up its guider profile too.** Every ARA profile owns a dedicated PHD2 profile on the guider daemon (created automatically on first guider connect), holding that rig's calibration and dark library. Deleting the ARA profile used to orphan its guider twin — profiles and dark files accumulating on the daemon forever. The delete now asks the connected guider to remove the twin, dark files included; if no guider is connected at the time, the cleanup is skipped with a log note (the ARA delete itself always succeeds either way). One known edge: renaming a profile and deleting it *without a guider connect in between* computes the twin's name from the new profile name, so the old-named twin can survive — the rename-sync hook that closes this is tracked in the ROADMAP.
- **§37.5 — the setup wizard now covers the unattended-failure shutdown.** The daemon has long known how to put a rig to bed when a failure suspends the run and nobody responds (park, warm the cooler, disconnect — the §58.12 countdown, configurable in Settings → Safety), but the wizard's safety screen never asked about it, so a fresh setup left it at defaults sight-unseen. Screen 15 now carries the on/off choice and the countdown minutes — the playbook's "WILMA offline: auto-abort after N min" line — with blank keeping the profile default as everywhere else in the wizard.
- **§37.5 — Tonight's Sky now warns when a target stays low all night.** New profile setting (Settings → Safety → Site → Soft warning altitude, default 30°, also on the wizard's site screen): a target that never climbs above the mark tonight is still listed — the hard horizon floor already decides inclusion — but its "Why?" score breakdown carries a tag, because at low elevation you image through more atmosphere and seeing/extinction soften detail. Advisory only: it never hides or down-ranks a target, and 0 disables it.
- **§37.5 — a max-runtime ceiling for unattended runs.** New profile setting (Settings → Safety → Site → Max sequence runtime, also on the wizard's site screen): a sequence still running past the ceiling is stopped gracefully — exactly as if you pressed Stop, so the current instruction cancels cleanly and every end-of-run step (session close, checkpoints, events) runs as normal — with a notification explaining why. 0 (the default) means no limit. The cap is re-read every minute, so raising it mid-run applies without a restart. Unattended-rig insurance: a stuck loop or an over-ambitious plan can't keep the mount tracking into the morning.
- **§37 — the setup wizard can now shed an old rig's paths.** A new profile starts as a clone of your current one, and a blank wizard field deliberately means "keep the current value" — which meant re-running the wizard for a new rig could never get rid of the old rig's ASTAP binary path, star-database path, save directory, or filename template: blank kept them, and there was nothing else to press. Those four fields now carry a reset button that marks the field to take the daemon default on Save; typing anything afterwards un-marks it (what you type wins).
- **§36.8 — "What could I shoot with a different rig?" Tonight's Sky can now answer.** The daemon has accepted per-request optical-train and mosaic overrides since slice 4a, but WILMA had no way to send them — the ranking was always your active profile's rig at 1×1. The Tonight's Sky panel now carries a what-if button (the tune icon in the header): dial in a different focal length, reducer/barlow, sensor geometry, or an N×M mosaic layout, and the list re-ranks against that train — "would the Veil fill the frame as a 2×2 with the 0.7× reducer on?" without touching your profile. Blank fields keep the profile's values (override just the reducer if that's all that changes), the tune icon tints while overrides are active so a re-ranked list can't masquerade as your real rig, and Reset returns to the profile's optics. Overrides are a session what-if, not a setting — they reset with the app.
- **§42.5 — guider outages are on the record now too.** The guider runs its own fault flow (it isn't an Alpaca device), so a mid-session guider link drop — or the guider daemon reporting the guide camera disconnected — executed your on-guider-lost policy and notified, but never landed in the persistent fault log: the fault feed and the Image Library's per-session fault timeline were blind to the one failure mode that most reliably ruins subs. Guider faults now get a fault-log row the moment they're detected, the row records what the policy did about it (paused/skipped/aborted, or notify-only when nothing was running), and a successful reconnect marks it resolved — same lifecycle as every other device's faults.
- **§64 — Live View star circles now work on colour (OSC) cameras.** Starting Live View with `annotate: true` on a one-shot-colour camera used to silently ignore the flag — you got the normal colour preview with no circles and no way to tell "annotation punted" from "no stars found". The daemon now detects stars on a luminance plane built from the same debayer pass that renders the colour frame (the raw Bayer mosaic itself is useless to the detector — the CFA pattern reads as per-pixel noise), so the green HFR-sized rings land exactly on their stars in the colour preview, with the same downscale-survival guarantees as the mono overlay.
- **§36 — the framing overlay's rotation now rides into the run.** Dial a position angle on the Sky Atlas framing overlay and hit Create Run, and the run's go-to-target step is a Center and Rotate at exactly that angle — slew, plate-solve to centre, rotate to the framed orientation — instead of a blind slew that silently dropped the rotation you framed. The dial's untouched 0° keeps the plain slew (no plate-solver requirement sneaks into runs you didn't frame with rotation), and appending a rotated target to an open sequence carries the angle the same way.
- **§38 — Center and Rotate now actually rotates.** On a rig with a rotator, the sequence instruction used to fail loudly rather than risk silently mis-framing (rotation wasn't wired). It now runs NINA's full rotate-then-centre flow: plate-solve the current field, sync the rotator to the solved sky angle, rotate by the shortest equivalent path to the plan's position angle (a 180°-flipped frame is the same framing, so 350° → 172° is a 2° nudge, not 178°), re-solve until within the profile's rotation tolerance — then run the usual solve → sync → re-slew centring loop. Multi-night mosaics and reproducible framing on rotator rigs work from a saved plan; without a rotator the behaviour is unchanged (position angle preserved in the plan, centring alone — exactly what NINA does).
- **§42.6 — fault records now name the exact frames they ruined.** A resolved equipment fault (the mount dropped and reconnected, tracking recovered) now carries the list of frames whose exposure overlapped the outage — the frame in flight when the fault hit, everything captured during the window, and any exposure still running when it resolved — scoped to the sequence session the fault belonged to. The correlation runs the moment the fault is marked resolved (whether by the recovery ladder or by an observed reconnect), and the frame IDs land in the fault record the Image Library's per-session fault timeline and `GET /api/v1/faults` already surface. The morning-after loop is closed: "why are these three subs trailed?" → the fault row points at exactly those three subs.
- **§42.3 — a device that comes back after the reconnect ladder gave up is now re-adopted automatically.** The hot-reconnect ladder (immediately, then 5/15/30/60 s) covers transient blips, but a real outage — a USB hub off for ten minutes, an AlpacaBridge host rebooting — outlives it, and the device used to stay dead until you manually reconnected. After giving up, the daemon now lingers on a slow cadence (every two minutes, for up to an hour): while the device still reads as lost it keeps re-dispatching the reconnect, and the moment the device answers again — whether the daemon's own dispatch landed or you re-seated a cable — it's re-adopted: sequence runs the fault reaction left paused are resumed, the fault record gets a `readopted` entry, and an all-clear notification posts. A device you *deliberately* disconnect is left alone — the linger stands down the moment the state reads as a clean disconnect rather than a lost device.
- **§42.2 — a mount that keeps failing slews now gets parked, and a camera that keeps failing captures pauses the night.** A single failed operation is instruction-level news — the instruction retries, fails, and the sequence's error handling moves on. But the *same device* failing over and over is equipment-level news, and the daemon now acts on the fault matrix's escalation rows: three definite op failures (a stalled call or a device error — advisories like a switch read-back mismatch or cooling drift never count) on the same device within ten minutes escalates. For the mount that means active runs are aborted and the mount is parked (a mount repeatedly refusing slews may be against an obstruction — physical safety first); for the camera, active runs are paused at the next instruction boundary (a camera that can't capture is burning sky time for nothing). Peripherals (focuser, filter wheel, rotator, switch, dome) stay notify-only however often they fail, per their matrix rows. The escalation lands everywhere faults already do — an `escalated` action on the fault-action WS channel, `escalated:abort_and_park` / `escalated:pause_sequence` stamped onto the fault-log row, and an Error notification naming the pattern. Camera capture failures (exposure never ready, device error during expose/download) now also publish op faults, so the camera row actually has eyes.
- **§42.2 — a rotator that quietly slips off its commanded angle is now caught, nudged, and reported.** The daemon used to trust a rotator move that dispatched cleanly: a belt that skipped after settle, a driver that silently ignored the move, or a mechanism that stopped short left the field rotated with nothing said. The rotator's status poll now checks the reported angle against the last commanded target (in the same mechanical/sky domain the move addressed, measured as proper circular distance so 359.8° vs 0.1° is 0.3° of drift, not 359°) with a ±0.5° tolerance. Persistent drift first gets the same one-shot recovery as a stuck switch port — the exact move is re-issued once — and only if it's still adrift after that does a `ValueMismatch` fault fire (chip overlay, fault feed, fault log). Reads while the rotator is moving or settling never count, an unreadable angle never accuses, and a sync or reverse flip resets the expectation.
- **§42.6 — a fault feed on the Imaging tab, and a per-session fault timeline in the Image Library.** The fault record now has faces: a collapsible Faults panel sits under the diagnostics panel on the Imaging tab — its header rolls up the live standing-fault state (the same overlay the equipment chips blend), and expanding it shows the persisted history newest-first (device, fault kind, what the reaction engine did about it, when), refreshing live as faults fire and resolve, with paging into older history. And in the Image Library, any session that recorded faults gets an amber "N faults" badge on its card — tap it for that night's fault timeline, the morning-after answer to "why are some of these subs trailed?".
- **§42 — the equipment chips now show faults, not just connection state.** WILMA's top-bar device chips used to read only the polled connection status, so a mount that silently stopped tracking or a dew-heater port that ignored its command looked exactly as green as a healthy rig. The client now listens to the live `equipment.fault` channel and blends a standing-fault overlay into each chip, worst-of: a recovery-tracked fault (device dropped, tracking lost) holds the chip red until the daemon reports it recovered — or until the device actually reconnects, covering the case where the automatic ladder gave up and you fixed it by hand; a one-shot advisory (stalled op, switch read-back mismatch, cooling drift) turns the chip amber and ages out after ten minutes so last night's hiccup doesn't haunt the morning. The fault overlay can make a chip look worse than its connection state, never better. This also lands the client's `/api/v1/faults` history client (model + API), which the upcoming fault feed and per-session fault timeline build on.
- **§42.5 — every equipment fault is now on the record, not just on the wire.** Faults used to be live-only: an `equipment.fault` event and a notification at 3 AM, gone by morning unless you were watching. Every detected fault (disconnect, tracking lost, stalled op, switch read-back mismatch, cooling drift) now lands in a persistent `faults` table in the session database the moment it's detected, and the §42.3 reaction engine stamps its outcome onto the same row as the episode unfolds — `reconnecting`, then `recovered` (with the recovery time) or `gave_up:` plus what it did about it — with the sequence session that was running attributed to the row. `GET /api/v1/faults` reads the history newest-first (filter by device type, fault kind, session, or unresolved-only; `GET /api/v1/faults/{id}` for one), giving WILMA's upcoming per-session fault timeline (§42.6) its data source, and giving you a morning-after answer to "what happened overnight?".
- **§42.4 — WILMA now hears about the exact instruction that failed, the moment it fails.** A sequence instruction that fails mid-run (a device op that errored, an unknown instruction in an imported sequence, a failed validation) used to be visible only as a status you could diff out of the progress payload after the fact. The daemon now emits a `sequence.instruction_failed` event the moment a leaf turns FAILED — carrying the run's usual identifiers plus the failed instruction's index and name — exactly once per failed instruction, and always before the run's terminal event, so a live sequence view can pin the failure to its row in real time. Together with the equipment-fault channel (which covers device ops that degrade gracefully without failing the instruction), this closes the §42.4 loop: every failure surfaces on one channel or the other.
- **§42.4 — a jammed filter wheel, a stalled focuser, or a power port that ignored you is now a reported fault, not a buried log line.** Every long-running sequencer device operation (focuser move, filter change, rotator move, mount op, dome op, switch write) already raced its blocking call against a generous timeout and degraded gracefully — but the failure only went to the log. Those sites now also publish an equipment fault: a timed-out or never-settling op is a `stall_timeout` (the classic jammed-EFW and stuck-focuser cases), any other device error is an `op_error`, both broadcast to WILMA and notified. And switch writes are now verified end-to-end: Ara remembers what it commanded on each port and checks the device's read-back against it (tolerance configurable as `switch_value_tolerance_pct`, default 5% of the port's range — enough to catch a commanded-ON/read-OFF relay); a port that stays wrong past a settle window raises a `value_mismatch` fault plus a structured `switch.value_mismatch` event naming the port, the commanded value, and what it actually reads. Sequence aborts you asked for never publish faults, and each failed command reports once — a retried instruction that fails again is a new report, not a duplicate.
- **§42.2 — Ara now catches a mount that quietly stops tracking, and a cooler that can't hold its temperature.** Two silent failure modes that ruin subs without ever throwing an error are now detected from the status polling Ara already does. If the mount reports tracking off for three consecutive ticks with no command that explains it — no slew, no park, nothing the daemon or you asked for — Ara publishes a tracking-lost fault and the reaction engine re-enables tracking on the retry ladder (pausing the sequence per your `on_tracking_lost` policy if it won't come back). If the camera's cooler is on but the sensor sits more than 5 °C off set-point for over 5 minutes (the TEC lost headroom, a fan died, the power brick browned out), Ara raises a cooling-drift warning — advisory only, it never aborts, since you may deliberately image warmer. Both detectors fire once per episode, re-arm on recovery, and are careful about innocent explanations: slews, parks, commanded changes, transient read hiccups, and download-stalled status ticks all count for the device, not against it.
- **§42.2/§42.3 — Ara now notices dead equipment and gets it back.** Every connected Alpaca device (camera, mount, focuser, filter wheel, rotator, dome, switch, flat panel) is now health-probed once per 2-second status tick; three consecutive missed probes — a pulled USB cable, a crashed driver, a dropped Wi-Fi bridge — flip the device to **Error**, broadcast an `equipment.fault` event to WILMA, and hand the fault to the new reaction engine. The reaction follows your policy (**Settings → Safety**): for a lost camera or mount it immediately pauses any running sequence at the instruction boundary (no more exposing into a dead camera), then hot-reconnects on a 0/5/15/30/60-second ladder; success resumes exactly the runs it paused, exhaustion executes the terminal action — stay paused for a camera (`on_camera_lost`), abort + best-effort park for a mount (`on_mount_lost`). Peripherals reconnect quietly in the background and only warn if they stay gone. Every step is observable (`equipment.fault_action_taken` events + notifications), single blips never trip anything, and the whole ladder can be disabled (`hot_reconnect_enabled: false`) for rigs that prefer to fail fast. The guider is untouched — it already has its own recovery. An `on_tracking_lost` re-enable policy is wired in for the upcoming tracking-loss detector.
- **§59.3/§59.4 — Smart Focus now reads which SIDE of focus you're on, and knows your telescope.** Declare your optical design once (`telescope_type` in the autofocus settings: refractor / SCT / Mak / RC / Newtonian; the wizard screen follows) and Smart Focus upgrades itself twice. First, obstructed scopes get a straighter ruler: the predicted move rides the donut's outer diameter (linear in defocus) instead of HFR's curve — automatically verified against your own calibration data and silently reverted to HFR if it doesn't hold. Second, the one-frame shot can now resolve *direction*, not just distance: every calibration sweep records both sides of focus, and Ara learns which star-shape signatures (like the radial-profile skew a scope with spherical aberration shows) genuinely differ between the sides of YOUR rig — then reads shot 1's frame against them. A confident read goes straight to focus in 2 shots even when the old toward-last-best guess would have been wrong; an unconfident one (well-corrected optics, ambiguous frame) falls back to the guess, and a wrong read is absorbed by the existing reversal ladder — so it can only save shots, never cost extra. The `autofocus.shot_complete` event now carries the signed `predicted_offset` and `direction_source` (`classifier` or `heuristic`) so WILMA's live panel can show its reasoning.
- **§59.15 — Smart Focus calibration endpoints + smarter progress.** `GET /api/v1/autofocus/calibration` returns the stored calibration (raw sweep samples, when it was taken, at what focuser temperature, on which filter; 404 = not calibrated), and `POST /api/v1/autofocus/recalibrate` clears it so the next autofocus runs a full calibrating sweep. The autofocus job's progress bar is now mode-aware — a 2-shot Smart run reads ~2/3 done instead of 2/9 — and the surprising fallbacks (a diverged Smart run, a device fault mid-run) post a warning notification, while routine condition-driven ones (thin stars, heavy defocus) stay quiet.
- **§59.2 — Smart Focus: autofocus in 2-3 exposures instead of a full sweep.** Once your rig is calibrated (which happens automatically on any successful Classic autofocus), the next autofocus reads how far out of focus you are from ONE exposure and moves straight there: shot 1 measures the defocus and predicts the move, shot 2 verifies — typically done in well under a minute instead of the 9-probe, multi-minute V-curve. A near-miss gets one small trim shot (three exposures is the budget); a wrong direction guess reverses with half the distance. Every failure mode falls back to the Classic sweep automatically (§59.11) — too few stars, a frame more defocused than anything calibrated, a calibration more than 8 °C stale, or a diverging run (position restored first), so the worst case is exactly the old behavior plus a few cheap shots. The run announces itself over WebSocket (`autofocus.started` with the chosen mode, `autofocus.shot_complete` per shot, `autofocus.fallback_classic` with the reason) so WILMA can show the §59.12 live panel. Each probe of the Classic V-curve sweep already measures the field's star shapes; Ara now keeps those measurements as labelled calibration samples and, when the sweep succeeds, stores them in the active profile (`focus_calibration`, with the focuser temperature and filter) — no extra exposures, no new step. This is the learning half of Smart Focus: a calibrated rig will soon predict the focus move from a single frame instead of re-running a full sweep. Samples that can't rebuild a usable focus model are discarded rather than overwriting a good calibration, and profile sharing strips the section (a calibration is fitted to one specific rig, so a recipient's regenerates on their first sweep).
- **§59.10 — autofocus now checks your collimation, for free.** After a successful autofocus on an obstructed scope (SCT / Mak / RC / Newtonian), Ara measures how far the central-obstruction shadow sits off-centre in the donut stars it already captured — the direct signature of a tilted secondary — and turns it into a collimation health read. A confident verdict is broadcast to WILMA (`autofocus.collimation_verdict`: severity, offset as a % of donut diameter, and star count), and the actionable bands also raise a notification — a warning at 5–15% ("slight miscollimation"), critical above 15% ("recommended to collimate before continuing"). It's robust by construction: only near-centre stars count (off-axis coma would masquerade as miscollimation), each is normalised by its own donut size, and the offsets are vector-averaged so random per-star and seeing noise cancel while a real tilt adds up. Refractors and in-focus fields have no donut, so they simply produce no verdict — never a false alarm. The exact clock-position direction ("toward 4 o'clock") is a later refinement; this reports the magnitude.
- **§64 — Live View can circle detected stars (mono).** Start Live View with `annotate: true` and each frame runs the same star detector autofocus uses, drawing a green circle around every detected star (sized by its HFR) right on the preview — a quick focus/framing aid without leaving the live view. Off by default, so nothing changes unless you ask for it; a rich field is capped so the overlay stays readable. Mono only for now (colour/OSC frames render as before — annotating a raw Bayer mosaic would misplace the circles).
- **§63.8 — dark-library and defect-map builds now show a live progress bar.** Building a dark library or bad-pixel map through the guider is a multi-minute job that used to look frozen — Ara only told you it had started, then nothing until it finished. Ara now polls the daemon's build progress once a second while the build runs and streams `guider.dark_library.progress` / `guider.defect_map.progress` updates (which exposure and frame it's on, out of how many), so WILMA can show a real progress bar the whole way through. A hiccup in a single progress read is skipped silently and polling resumes — it never disturbs the build itself.
- **A dropped guide camera now pauses your sequence instead of shooting unguided.** When the guiding daemon reports the guide camera disconnected mid-session — e.g. its USB drops — Ara now runs your "when guiding is lost" policy (pause / skip target / abort, per **Settings → Safety → Policies**) rather than quietly continuing to expose without guiding, and alerts you even if no sequence is running. Because the daemon's own link is still up and it recovers the camera on its own, the guider session stays **connected** (no needless teardown) with wording that reflects that — it won't tell you to reconnect a guider that never disconnected. (Reacting to the daemon *giving up* on a stuck reconnect, and re-reacting to repeated drops in one session, are tracked follow-ups.)

### Added
- **§42.2 — a wrong-reading switch port gets one nudge before the alarm.** When a written switch port's read-back persistently disagrees with what was commanded (the dew-heater relay that didn't flip), Ara now re-issues the same command once — with a fresh settle window — before declaring the fault, matching the fault matrix's "Re-command → Notify" recovery step. A port the nudge fixes never bothers you; a port that stays wrong faults exactly once per command episode, as before.

### Fixed
- **§37.10 — the plate-solve engine picker no longer offers engines the daemon can't use.** The daemon ships ASTAP only; picking astrometry.net or PlateSolve 2 quietly solved with ASTAP anyway. The dropdown now lists ASTAP alone — a profile still carrying one of the old values (say, imported from NINA) shows it honestly labeled "not supported — ASTAP is used" until you switch it.
- **§63.8 — a dead progress feed during a dark-library build is no longer silent.** While a dark-library or defect-map build runs, the daemon polls the guider for progress once a second and deliberately shrugs off individual failures (a dropped tick is cosmetic). But when the progress RPC is persistently unreachable — an older guider daemon without it, a firewall, a mid-build network fault — every failure was swallowed at debug level while the client sat on an indeterminate progress bar with no clue why. The build log now carries one warning per build (never a page of repeats): after five consecutive failed polls, or at build end when not a single poll succeeded. The build itself is untouched either way — it never depended on the progress feed.
- **§30 — reconnecting an older rig now actually makes it the active server.** With more than one server saved (observatory + travel rig), the app always talked to the most-recently-*saved* one: re-confirming a server WILMA already knew was a silent no-op, so the "last one used is the default" promise was broken — you could be pointed at the wrong daemon with no way to switch short of clearing storage. Re-confirming a known server now moves it to the front of the line (and refreshes its stored version/mDNS metadata), and every per-server API in the app routes through one canonical active-server definition instead of each call site reaching into the saved list on its own — so a future explicit server switcher (§55.1) changes one line, not thirty call sites.
- **§64/§59 — star detection with a cap no longer measures the whole field first.** Every consumer that caps its star count (the Live View overlay's 250-marker cap, autofocus's bright-star selection) used to pay the full per-star measurement for every blob in the frame and then throw away all but the brightest N — on a dense Milky Way field, thousands of centroid/HFR measurements per annotated live frame for 250 kept markers. The detector now measures brightest-first and stops when the cap is filled, picking the identical star set (a saturated reject still hands its slot to the next-brightest, as before). Uncapped detection is unchanged.
- **§64 — an annotated Live View no longer delays your real captures.** Rendering a live frame — and, with star circles on, the full-frame star-detection pass, real CPU time on a big sensor — used to run while Live View still held the camera's capture slot, so every annotated preview frame extended the window a manual or sequence capture had to wait. The slot is now released the moment the frame's pixels are downloaded; the stretch/debayer/detection/JPEG work happens after, and a capture can slip in while the preview renders.
- **Plate-solver crashes are no longer indistinguishable from "no solution".** The solver process's exit code was never checked — a crashed ASTAP looked identical to a clean not-solved. Non-zero exit codes are now logged with their documented meaning (for ASTAP: no solution, too few stars, image read error, missing/corrupt star database — the last of which no amount of per-solve retrying fixes, and now says so), so a failing solve chain is diagnosable from the log.
- **§42.2 — a failed mount slew, focuser move, filter change, rotator move, switch write, or dome op now actually fails the instruction.** These operations already raced their device calls against generous timeouts and reported faults to WILMA — but at the sequence level the failure was silently swallowed: the instruction carried on as if the op had succeeded, so a jammed filter wheel could quietly shoot a whole night through the wrong filter, and a slew that never completed could expose on the wrong field. Every one of these operations now fails its instruction loudly on a definite failure (stalled call, device error, or an op that dispatched but never reached its terminal state), which finally engages the per-instruction retry count, the live `sequence.instruction_failed` event, and the sequence's configured error behavior. Deliberate aborts still cancel cleanly (never reported as faults), device disconnects keep their own reconnect-ladder handling, and a filter restore after plate-solve centering stays best-effort so a wedged wheel can't mask the centering result it was cleaning up after.
- **The fault log now notices when you fix things yourself.** A disconnect fault whose automatic recovery ladder gave up stayed "unresolved" in the fault history forever — even after you walked out, reseated the cable, and reconnected the device — because only the ladder's own success ever marked a fault resolved. The daemon now stamps the resolution the moment a device reports connected again, whatever healed it, so the history (and everything reading it, like the fault feed and the chips' launch-time seeding) tells the truth about what's actually still broken. The recorded reaction outcome is kept as history, and a reconnect only closes disconnect faults — it doesn't retract a tracking-loss or an advisory.
- **Paginated lists no longer cut off after the first page.** The notifications list, diagnostics history, and image-library frames list all reported "no more pages" regardless of how many rows actually existed — a client paging through its notification inbox or a long session's frames stopped after page 1 and silently never saw the rest. The paging loop consumed the look-ahead row that signals "there's another page" before checking it; all three queries now check the page boundary first, so `has_more` and the next-page cursor are returned correctly (with regression tests walking a multi-page catalog end-to-end for each).
- **Star detection is more reliable on vignetted frames.** The background level that sets the star-detection threshold was estimated from a linear stride through the image memory, which on many frame sizes aliases onto just a few fixed columns — so a vignette or other left-to-right brightness variation could bias the estimate and, at worst, make detection miss stars or over-trigger. The background is now sampled on an even 2D grid across the whole field, so the threshold reflects the real sky background regardless of how brightness varies across the frame. This feeds everything built on star detection: autofocus HFR, live-view star stats, and plate-solving.
- **The guider is now correctly identified as openastro-guider.** The guiding daemon was renamed `openastro-phd2` → `openastro-guider`, but Ara's connect-time check still matched only the old name — so against the current daemon it would fall back to "connected to upstream PHD2" and warn that headless guiding features might be unavailable, even on a fully-supported rig. Ara now runs the daemon's `get_version` handshake on connect and reads its fork marker (accepting the old name too), so a supported guider is recognized as such and the spurious warning is gone.

Below-horizon shading, a planning-time picker, and a custom terrain horizon follow in later slices.

(Aladin is an equatorial atlas, not an alt/az planetarium, so the horizon is a curve that moves with location and time.) The client horizon overlay + time control build on this in follow-up slices.

### Fixed
- **Equipment status no longer shows stale "connected" green after the server drops.** The top-bar device chips (CAM/FW/FOC/MOUNT/ROT/GUIDE/FLAT/SW/WX/SAFE/DOME) and the device panels in Options mapped each device's *last-known* status straight to the dot/layout, so when the server was killed or became unreachable they kept rendering a false green long after the app had lost contact. A new `serverLinkUpProvider` (true only while the live server link is fully connected) now gates them: link down → chips go grey (disconnected) and each device panel shows a clear "Server disconnected — device status is unavailable" state instead of stale connect/disconnect controls. They repopulate with real status the moment the link is back.
- **§60.9 — the live server WebSocket no longer drops every fresh connection (status was stuck "Reconnecting").** Two compounding bugs: (1) the daemon's resume phase enforced its 5-second "send a resume request" window by cancelling its `ReceiveAsync`, but a cancelled `WebSocket.ReceiveAsync` *aborts* the socket (close 1006) — so any client that stayed silent past 5s (i.e. every fresh subscriber) was dropped, then reconnected and dropped again, forever. The resume phase now races the receive against a delay *without* cancelling it, handing the still-pending receive to the main loop on timeout, so a silent client stays connected as a fresh subscription. (2) The client only sent a resume frame on a *re*connect; a fresh connect sent nothing, so it tripped (1) every time. The client now always sends a resume frame first (token `"0"` when fresh), which also draws a resume-response back that flips the link to connected. The status chip is also relabelled to the explicit client↔server link — **Server connecting… / connected / reconnecting… / disconnected** — so it can't be mistaken for a device/camera state.
- **Mount manual-control: a held direction button now stops if the mount goes busy.** On the §37.5 direction pad, if the mount became busy from another source (a sequence step, a GoTo) while a direction was physically held, the button greyed out but the axis kept moving until the user released. The press-and-hold button now dispatches the stop on the disable transition (not only on release/teardown), so a held nudge halts as soon as the control disables. (Stop / AbortSlew remained the deadman regardless.)
- **Mount Park now works (it was silently failing).** Two issues: (1) the Park button POSTs `/equipment/mount/park` with no body, but the endpoint required a `[FromBody] ParkRequestDto`, so model binding rejected the bodyless request (400/415) and the mount never parked — the body is now optional (its only field is an optional reason, defaults to null). (2) Some mounts (e.g. iOptron) won't execute a park from a stationary state, so the daemon now enables tracking before issuing Park — exactly as ASCOM ConformU/NINA do. (Unpark/Stop were unaffected; Slew always sends RA/Dec.) Note: a mount that parks to a position *other* than its home/zero may still no-op a park issued from the home position — slew it to a normal pointing position first (see the new Manual control GoTo).
- **Every device panel now offers Reconnect again after you disconnect it.** The device services keep the last device after a disconnect and report it with `state == disconnected` (a non-null status GET, not a 404), but the shared `EquipmentConnectionCard` only showed its disconnected layout — the one carrying the **Reconnect** + **Connect…** buttons — when the status was null. So a device disconnected during the session rendered the *connected* layout (just a Disconnect button), with no way to reconnect it short of re-discovering via Connect…. The card now treats a `disconnected` status the same as a missing one, so Reconnect reappears the moment you disconnect — across all nine equipment panels.
- **The top-bar FLAT chip now actually turns green when you connect a flat panel.** The chip was wired to the live flat-panel provider, but the Flat Panel settings panel was the one device panel that still connected through the old wizard-style row (`AlpacaDeviceRow` on `equipmentConnectionProvider`) instead of through that provider — so connecting a flat never told the chip's provider, which had stopped polling after its initial "disconnected" read, and the dot stayed grey. The Flat Panel panel now uses the same shared `EquipmentConnectionCard` as the other eight device panels (connect/disconnect/reconnect on the live `flatPanelProvider`), with a live cover/light readout; connecting — or turning the calibrator light on — now lights the chip and shows the state in the panel. This also gives the flat panel the same disconnected-state **Reconnect** affordance as every other device.
- **§52.1 — switches now auto-reconnect on boot like the rest of your gear.** Switch was the one device type the daemon never re-established on startup: the selection store only remembered one device per type (so a multi-switch rig couldn't round-trip), there was no Switch case in the auto-connect service, and no Switch auto-connect-on-boot toggle. Now every connected switch is remembered independently (by its Alpaca device number), the auto-connect service reconnects them all on boot, and the Switch panel has an **Auto-connect on boot** toggle (default on). So after a daemon/rig restart your switches come back without manual reconnection — matching camera/mount/focuser/etc.
- **The top-bar FLAT (cover calibrator / light panel) chip now shows live status.** It was a hardcoded placeholder that always rendered disconnected — so a connected light panel never lit up at the top. It now watches the flat-panel status provider like the other equipment chips: green when connected, amber while the cover is moving, and routes to the Flat Panel settings panel on tap. (The daemon already fully supported CoverCalibrator/flat devices — discovery, connect, control, auto-connect; only the client chip was static.)
- **§36 — the macOS client now actually runs CEF 149 multi-process, and the native macOS build is no longer broken.** The preceding Sky Atlas change raised the client to **C++20** and documented CEF 149, but left the `webview_cef` submodule pinned at **CEF 130** — whose `libcef_dll_wrapper` doesn't provide the wide-char stream operators (`operator<<(ostream&, char16_t/wchar_t const*)`) the C++20 build links against, so `flutter build macos` failed with *undefined symbols* (client CI runs `analyze + test` only and never compiles the native Runner, so the break slipped through the gate). The submodule is bumped to the CEF 149 multi-process build, and the macOS Runner now embeds the **full five-helper set** (base + `GPU`/`Renderer`/`Plugin`/`Alerts`) Chromium-on-macOS requires — a single helper made the GPU subprocess abort with `gpu_process_host error_code=1003`. - **§6 — the Switch panel stayed stuck on "Connecting" after Add switch (client).** The daemon's switch connect is 202-accepted + runs in the background, so a freshly-added switch first reads back as `connecting`; the panel re-read the list once right after the connect and then never again (the provider is pull-on-demand), so the card sat on **Connecting** until the user re-opened the panel — looking like the connect failed even though the daemon had connected. - **§6 — equipment discovery parsed every device to blanks (client).** `DiscoveredDevice.fromJson` read **camelCase** keys (`uniqueId`, `hostName`, `alpacaDeviceNumber`, …) with a comment claiming the daemon emitted camelCase, but the daemon serializes with `SnakeCaseLower` (so the wire is `unique_id`, `host_name`, `alpaca_device_number`, …). Every discovered device therefore came back with a blank name + empty `uniqueId` + device number `0`, breaking the whole discover→select→connect flow for **all** equipment types (the chooser showed nameless rows; a "connection" pointed at a default device). - **§29.9 — `GET /api/v1/server/logs/download` no longer 500s on the active log.** The download streamed the live log file while the Serilog sink kept appending to it; `Results.Stream` set `Content-Length` from the file size at open time, so the copy ran past that length (the file grew mid-flight) and Kestrel aborted with `Response Content-Length mismatch: too many bytes written`. - **§38 — the sequencer toolbar no longer offers an inoperative Pause control.** The daemon's sequence `PauseAsync` is an accepted no-op — the headless execution engine has no pause hook yet, so it deliberately never suspends a run or emits a `paused` event. The toolbar nonetheless showed an enabled **Pause** button (and a dead Run→Resume relabel) while a run was active, so clicking it returned `202 Accepted` but did nothing, with the run-state never reporting `Paused` — misleading the user into thinking the run had paused. Pause/Resume are now removed from the toolbar until the engine grows real run suspension; Run / Abort are unchanged. The run-state model (including the `paused`/`Paused` state) is retained so Pause/Resume can be re-surfaced wholesale when the daemon supports it.

### Removed
- **The dead `VVVV.FreeImage` dependency (and its NU1701 build warning).** `OpenAstroAra.Image` still referenced `VVVV.FreeImage` (a .NETFramework-only package), but nothing used it: the RAW-converter that did was deleted during the port, the §2105 render path is SkiaSharp, and only a vestigial enum label remained. Because it's a framework package consumed on `net10.0`, NuGet emitted **NU1701** on every project that transitively referenced Image (Image/Server/Equipment/PlateSolving/Sequencer) — 5 warnings on every build, previously allow-listed in `Directory.Build.props` (`WarningsNotAsErrors`) to avoid failing the build. Removing the unused `PackageReference` drops a shipped-but-unused native lib from the daemon bundle, makes `NOTICE.md` accurate (FreeImage is no longer distributed), and yields a clean build with **no NU1701**; the now-unnecessary NU1701 allow-list entry is removed so a *new* framework-package dependency surfaces as an error and gets a conscious decision rather than slipping in silently. (One unrelated `NU5104` — Accord.Math prerelease — remains allow-listed.)
- **§36 — the broken `horizon-default` Data Manager download.** It pointed at the dead `data.openastro.net` host (so the download always failed) and was miscategorised: a site horizon — a flat default or a survey — is generated **locally** for the §36 Tonight's-Sky overlay (Meeus altitude formula + below-horizon shading), not fetched as a Data Manager package. Removing it means no broken download ships; the real horizon feature re-introduces horizon data on the Tonight's-Sky side when it's built (tracked in `design/PORT_TODO.md`). The Data Manager's catalog + per-package integrity digests are now injectable into `DataManagerService` (default = the production set), so the download-worker tests run against a self-contained synthetic catalog instead of depending on the production catalog's exact contents.
- **§68.1 — the AlpacaBridge version gate.** The daemon used to probe an AlpacaBridge `GET /version` and gate every equipment connect on it: too-old bridges were blocked outright (503) and "warn-band" bridges connected but raised an advisory (an `equipment.alpaca_bridge_outdated_warn` WS event + a banner on the client's Equipment panels). This was non-conformant — ASCOM Alpaca is open by design and has no such handshake or `/version` endpoint (AlpacaBridge doesn't even serve one), so the probe always failed and the gate was pure friction. Removed end-to-end: the version probe, handshake service, gate, and gate notifier; the 503/warn-band WS event (`WsEventCatalog.EquipmentAlpacaBridgeOutdatedWarn`); the client `AlpacaBridgeWarningBanner` + its `alpaca_bridge_warning_state`; and the gate wiring on every REST `/connect` handler and the boot auto-connect path. Equipment connects now go straight through to the Alpaca client. (Playbook §68 to be reconciled by the maintainer — flagged in `design/PORT_TODO.md`.)
- **§29.9 — `PlaceholderLogService`.** Removed now that the real `LogService` is wired (see Added) — it returned fixture log entries and a 404 download.
- **§54 — `PlaceholderBugReportService`.** Removed now that the real `BugReportService` is wired (see Added) — it returned a synthetic "ready" record and a 404 download.
- **§38 — the superseded read-only sequence display tree.** Deletes `SequenceTree`, `InstructionEditor`, and the `SequenceController`/`selectedNodeIdProvider` state (`state/sequencer/sequence_state.dart`). The Sequencer tab now renders the editable §38 surface (`SequenceEditorTree`/palette/field editor over `sequenceEditorProvider`), so the old lossy `SequenceNode` display widgets had no remaining users. `SequenceNode`/`parseNinaSequenceBody` stay — they still back `getSequence` and the editor tree's depth cap.

### Security
- **CVE-2025-6965 — bundled SQLite updated to a patched build.** `Microsoft.Data.Sqlite` / `Microsoft.EntityFrameworkCore.Sqlite 10.0.x` transitively pull the native `SQLitePCLRaw.lib.e_sqlite3 2.1.11`, which bundles SQLite older than 3.50.2 — affected by CVE-2025-6965 (CWE-197 numeric truncation, possible memory corruption; CVSS 7.2). The daemon now pins `SQLitePCLRaw.lib.e_sqlite3 3.50.3` (SQLite 3.50.3, patched) over the transitive version; the native SQLite C API is backward-compatible with the existing managed provider, so this swaps in fixed SQLite without changing the data layer. The pin can be dropped once `Microsoft.Data.Sqlite` ships a build on the patched lib.

### Fixed
- **§28/§50 (v0.1.0) — Recovered-frame capture times are read as UTC, not local.** When the daemon recovers an orphaned FITS frame into the catalog, it reads the capture time from the file's `DATE-OBS` header. FITS defines that value as UTC, but it's usually written without a timezone suffix — and the parser was treating a suffix-less value as **local time**, shifting the stored `captured_utc` by the machine's UTC offset (and stamping it with a non-UTC offset that would have mis-sorted it in the stats time-window queries). A zoneless `DATE-OBS` is now correctly read as UTC, and an explicitly-offset one is converted to UTC, so recovered frames land at the right instant and sort correctly. (No effect on frames captured live through the sequencer — only the orphan-recovery path.)
- **§43-2 (v0.1.0) — Backups dir self-heals crash-only orphan archives at startup.** If the daemon was hard-killed (power loss / SIGKILL) mid-backup, two kinds of dead file could linger under `backups/` forever — a `.tmp-*.zip` staged before its atomic reveal, or a `backup-*.zip` whose `.meta.json` manifest never got written. The snapshot list already hid both, but nothing reclaimed the disk. The daemon now sweeps them on boot (a graceful create still cleans up after itself, and real manifest-backed snapshots are never touched), mirroring the §36-2c sky-data scratch sweep.
- **§55.1 (v0.1.0) — Settings-sync store self-heals orphaned temp files at startup.** The WILMA settings-sync store writes `client-settings.json` atomically via a `.tmp-*` sibling + rename; a hard kill in the window between the write and the rename could strand the temp file. The daemon now reclaims any such orphans on boot (the live settings file is never touched), the same boot-time sweep applied to backups.
- **§50 (v0.1.0) — Stats time-window filters normalize the cutoff to UTC.** The stats queries that take a `since` cutoff (Focus & Temperature, Guiding RMS) and the Calendar's rolling 30-day window now normalize the bound to UTC before comparing it against the stored UTC capture timestamps. A caller passing a non-UTC `DateTimeOffset` would otherwise have filtered against the wrong instant (the comparison is lexicographic on ISO strings). Latent today — all current callers already pass UTC — but it hardens every `since`-taking query via one shared helper.
- **§50.19 (v0.1.0) — Stats Achievements no longer flashes an error if a refresh fails exactly as you switch servers.** The Achievements section had its own copy of the refresh/generation-guard logic (it predated the shared one) that was missing a later fix: a refresh whose fetch *failed* right as the active server changed would rethrow a now-irrelevant error and could briefly flash a stale banner over the new server's records. Achievements now uses the same `StatsRefreshMixin` as the other seven Stats sections, which discards that stale failure — bringing all eight to identical, tested behavior.
- **§63 — A guider RPC call can no longer hang the connect forever on a silent/misbehaving guider.** `PHD2Guider.SendMessage` read the JSON-RPC response with `ReadLineAsync()`, but `TcpClient.ReceiveTimeout` only governs *synchronous* reads — so if the guider never returned a response with the expected id (a half-broken or wrong-version guider), that call and the whole connect handshake would block indefinitely. The async read is now bounded by the request's `receiveTimeout`, so a non-responsive call fails and surfaces instead of stalling. Surfaced by the §42.2 virtual-observatory bench.
- **§63 — Guider event listener no longer busy-polls the OS TCP table (could crash on macOS).** The guider's PHD2 event-stream reader (`PHD2Guider.RunListener`) previously detected a dropped connection by enumerating the OS active-TCP-connection table (`GetActiveTcpConnections()`) every 500 ms and also spun on `NetworkStream.DataAvailable`. That table enumeration could throw on macOS (null/duplicate local endpoints), tearing the listener down right after connect, and the 500 ms poll added latency + CPU churn. It's now a proper async line reader — events arrive as soon as the guider sends them, and end-of-stream (the guider closing the socket) is the disconnect signal that drives §63.3 crash-recovery. No behavior change against a real guider; removes a macOS-only failure path and the busy-loop.
