# Changelog

All notable changes to OpenAstro Ara are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).
Versioning follows [Semantic Versioning](https://semver.org/spec/v2.0.0.html) —
pre-1.0, breaking changes can happen in any release per the playbook §0.6.

For Ara-specific release tagging convention (`v0.0.1-ara.N`), see
[design/PORT_PLAYBOOK.md §34.5](design/PORT_PLAYBOOK.md). Phase boundary tags
(`phase-N-complete`) are NOT release tags — they're internal milestone markers
used by the port's `port/ara → master` promotion cadence (playbook §22.0).

## How to update this file

Every PR that lands a user-visible change adds a bullet to the **[Unreleased]**
section as part of its diff (convention enforced by the port-driver skill, see
`.claude/skills/port-driver/SKILL.md` step 4). Pick the matching subsection:

- **Added** for new features, endpoints, settings, or capabilities.
- **Changed** for changes in existing behavior.
- **Deprecated** for soon-to-be removed features.
- **Removed** for now-removed features or files.
- **Fixed** for bug fixes.
- **Security** for vulnerability fixes.

Mechanical PRs (pure renames, pure deletes during demolition, doc-only edits to
internal design files) don't need a CHANGELOG entry. When in doubt, add one.

At release time (e.g. `v0.0.1-ara.1`), the **[Unreleased]** section is renamed
to `[0.0.1-ara.1] - YYYY-MM-DD` and a fresh `[Unreleased]` placeholder is added
at the top. This happens in the same commit that pushes the release tag.

### Fixed
- **§28/§50 (v0.1.0) — Recovered-frame capture times are read as UTC, not local.** When the daemon recovers an orphaned FITS frame into the catalog, it reads the capture time from the file's `DATE-OBS` header. FITS defines that value as UTC, but it's usually written without a timezone suffix — and the parser was treating a suffix-less value as **local time**, shifting the stored `captured_utc` by the machine's UTC offset (and stamping it with a non-UTC offset that would have mis-sorted it in the stats time-window queries). A zoneless `DATE-OBS` is now correctly read as UTC, and an explicitly-offset one is converted to UTC, so recovered frames land at the right instant and sort correctly. (No effect on frames captured live through the sequencer — only the orphan-recovery path.)
- **§43-2 (v0.1.0) — Backups dir self-heals crash-only orphan archives at startup.** If the daemon was hard-killed (power loss / SIGKILL) mid-backup, two kinds of dead file could linger under `backups/` forever — a `.tmp-*.zip` staged before its atomic reveal, or a `backup-*.zip` whose `.meta.json` manifest never got written. The snapshot list already hid both, but nothing reclaimed the disk. The daemon now sweeps them on boot (a graceful create still cleans up after itself, and real manifest-backed snapshots are never touched), mirroring the §36-2c sky-data scratch sweep.
- **§55.1 (v0.1.0) — Settings-sync store self-heals orphaned temp files at startup.** The WILMA settings-sync store writes `client-settings.json` atomically via a `.tmp-*` sibling + rename; a hard kill in the window between the write and the rename could strand the temp file. The daemon now reclaims any such orphans on boot (the live settings file is never touched), the same boot-time sweep applied to backups.
- **§50 (v0.1.0) — Stats time-window filters normalize the cutoff to UTC.** The stats queries that take a `since` cutoff (Focus & Temperature, Guiding RMS) and the Calendar's rolling 30-day window now normalize the bound to UTC before comparing it against the stored UTC capture timestamps. A caller passing a non-UTC `DateTimeOffset` would otherwise have filtered against the wrong instant (the comparison is lexicographic on ISO strings). Latent today — all current callers already pass UTC — but it hardens every `since`-taking query via one shared helper.
- **§50.19 (v0.1.0) — Stats Achievements no longer flashes an error if a refresh fails exactly as you switch servers.** The Achievements section had its own copy of the refresh/generation-guard logic (it predated the shared one) that was missing a later fix: a refresh whose fetch *failed* right as the active server changed would rethrow a now-irrelevant error and could briefly flash a stale banner over the new server's records. Achievements now uses the same `StatsRefreshMixin` as the other seven Stats sections, which discards that stale failure — bringing all eight to identical, tested behavior.
- **§63 — A guider RPC call can no longer hang the connect forever on a silent/misbehaving guider.** `PHD2Guider.SendMessage` read the JSON-RPC response with `ReadLineAsync()`, but `TcpClient.ReceiveTimeout` only governs *synchronous* reads — so if the guider never returned a response with the expected id (a half-broken or wrong-version guider), that call and the whole connect handshake would block indefinitely. The async read is now bounded by the request's `receiveTimeout`, so a non-responsive call fails and surfaces instead of stalling. Surfaced by the §42.2 virtual-observatory bench.
- **§63 — Guider event listener no longer busy-polls the OS TCP table (could crash on macOS).** The guider's PHD2 event-stream reader (`PHD2Guider.RunListener`) previously detected a dropped connection by enumerating the OS active-TCP-connection table (`GetActiveTcpConnections()`) every 500 ms and also spun on `NetworkStream.DataAvailable`. That table enumeration could throw on macOS (null/duplicate local endpoints), tearing the listener down right after connect, and the 500 ms poll added latency + CPU churn. It's now a proper async line reader — events arrive as soon as the guider sends them, and end-of-stream (the guider closing the socket) is the disconnect signal that drives §63.3 crash-recovery. No behavior change against a real guider; removes a macOS-only failure path and the busy-loop.

### Changed
- **§36 (v0.1.0) — macOS Sky Atlas renders; webview_cef on CEF 130 + Swift Package Manager, vendored as a submodule.** The embedded Aladin atlas (Planning tab) now renders on macOS. The `open-astro/webview_cef` fork was modernized — CEF 103 → **130.1.2** (matching Linux), **Swift Package Manager** support (resolving Flutter 3.44+'s "does not support Swift Package Manager" warning; CEF ships as `.xcframework` binary targets), and offscreen **WebGL via SwiftShader** (windowless CEF previously passed `--disable-gpu`, which rendered Aladin Lite v3 as a black rectangle). The client now vendors the fork as a git submodule at `client/openastroara_client/packages/webview_cef` (path dependency) instead of a git-SHA pin; the macOS Runner project moved from CocoaPods to Swift Package Manager (webview_cef was the only pod). The CEF binaries are git-ignored — run `packages/webview_cef/macos/setup_cef.sh` once after checkout (`git submodule update --init`) to stage them before a macOS build. Windows stays on CEF 101 pending a separate migration (open-astro/webview_cef#2).
- **§43-2b (v0.1.0) — Backup restore screen waits for the real outcome.** The Backup & Restore screen now polls the daemon's `clone-status` after starting a restore instead of reporting "Restore complete" the instant the request is accepted. It shows the actual result once the background worker finishes — including a **worker-side failure** (e.g. an extract/swap error) that the 202 response couldn't have revealed — and times out gracefully if the restore never reports a terminal state. Pairs with the §43-2b(a) server worker.
- **§43-2b (v0.1.0) — Backup restore runs on a background worker with live progress.** Restoring a snapshot (`POST /api/v1/backup/restore-zip`) now returns `202 Accepted` immediately and performs the extract + atomic swap on a background worker; `GET /api/v1/backup/clone-status` reports the worker's live state (`idle` → `running` → `done`/`failed`) instead of always `idle`. Validation that can fail fast (unknown snapshot → 404, corrupt-archive checksum → 422, unsupported source → 422) still happens synchronously before the 202, and a second restore while one is already running is refused with **409 Conflict** (one restore at a time — both mutate the live profile). The atomic all-or-nothing rollback behaviour is unchanged. (Still deferred to later §43-2b slices: remote/cloud restore sources and the frame-metadata/logs areas.)
- **§50 (v0.1.0) — Stats timestamp parsing is unified + hardened.** All six §50 stats models now share one `parseStatsUtc()` helper for wire timestamps. Four of them (Overview, Targets, Best Frames, Achievements) previously used a form that would mis-shift a timezone-less timestamp by the client's local offset; they now reinterpret it as UTC like the daemon intends. No user-visible change today (the daemon always sends offset-bearing timestamps), but it removes the latent drift and de-duplicates six copies into one tested helper.
- **§50 (v0.1.0) — Stats queries get a light-frame index.** Added a partial covering index `idx_frames_light_captured` on `frames(captured_utc) WHERE frame_type = 'light'`, so the Stats dashboard's light-frame, time-ordered queries (targets, calendar, frame-quality, best-frames, focus-temp) stop treating the `frame_type = 'light'` filter as a residual scan over the full catalog. Additive + idempotent — a database from an earlier build picks it up at next startup; no behavior change, just faster stats as the frame catalog grows.
- **§50 — Stats refresh keeps your data on screen (whole dashboard).** Manually refreshing any Stats section — Overview, Targets, Best Frames, and the Frame Quality / Guiding RMS / Calendar charts — no longer blanks it to a "Loading…" placeholder and back; the last-loaded data stays visible while the header spinner runs, and if the refresh fails the section keeps showing it (tile sections with a subtle "Couldn’t refresh — showing the last loaded …" banner, charts with a small "Stale" overlay chip) instead of dropping to a full error/retry screen. All six sections now share a `StatsRefreshMixin` (the §50.19 Achievements section's pattern), which also correctly discards a stale result — success or failure — when the active server changes mid-refresh.
- **§63.1 — The daemon connects to the guider as a service instead of launching it.** When you connect the guider, ARA core now connects to the already-running `openastro-guider`/PHD2 service on its host/port (default `localhost:4400`) — the normal headless deployment where the Pi boots the guider, ARA core, ASTAP, and the Alpaca bridge as services at power-on. If the guider isn't reachable yet, ARA asks the service manager to start it (`systemctl start`, via the same supervisor that handles §63.3 crash-recovery) and waits briefly, rather than spawning a local PHD2 process. The inherited NINA-desktop behavior — launching `phd2.exe` and waiting for its GUI window (`WaitForInputIdle`) — has been retired; it could not work on a headless daemon and blocked connecting to a guider running on localhost. Off-systemd (dev/CI) the start request is a no-op, so ARA simply connects to whatever guider is already listening.

- **§43-2a (v0.1.0) — Backup: restore a snapshot.** `POST /api/v1/backup/restore-zip` now actually restores — it validates the chosen snapshot against its stored SHA-256, then rolls the selected areas (`restore_profiles` → `profile.json`, `restore_sequences` → the `sequences/` tree) back to that snapshot.
