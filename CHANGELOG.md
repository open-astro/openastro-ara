# Changelog

All notable changes to OpenAstro Ara are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).
Versioning follows [Semantic Versioning](https://semver.org/spec/v2.0.0.html) ‚Äî
pre-1.0, breaking changes can happen in any release per the playbook ¬ß0.6.

For Ara-specific release tagging convention (`v0.0.1-ara.N`), see
[design/PORT_PLAYBOOK.md ¬ß34.5](design/PORT_PLAYBOOK.md). Phase boundary tags
(`phase-N-complete`) are NOT release tags ‚Äî they're internal milestone markers
used by the port's `port/ara ‚Üí master` promotion cadence (playbook ¬ß22.0).

## How to update this file

Every PR that lands a user-visible change adds a bullet to the **[Unreleased]**
section as part of its diff (convention enforced by the port-driver skill, see
`.claude/skills/port-driver/SKILL.md` step 4). Pick the matching subsection:

- **Added** for new features, endpoints, settings, or capabilities.
- **Changed** for changes in existing behavior.
- **Deprecated** for soon-to-be removed features.
- **Removed** for now-removed features or files.
- **Fixed** for bug fixes.
- **Security** for vulnerability fixes.

Mechanical PRs (pure renames, pure deletes during demolition, doc-only edits to
internal design files) don't need a CHANGELOG entry. When in doubt, add one.

At release time (e.g. `v0.0.1-ara.1`), the **[Unreleased]** section is renamed
to `[0.0.1-ara.1] - YYYY-MM-DD` and a fresh `[Unreleased]` placeholder is added
at the top. This happens in the same commit that pushes the release tag.

- **¬ß58.4 ‚Äî Automatic meridian flip.** When a target crosses the meridian mid-sequence, the daemon now runs the full flip recovery automatically: it stops guiding, waits out the configured pause window, flips the mount, plate-solves and re-centers on the exact target, then resumes guiding and settles ‚Äî and verifies the mount's pier side afterward. A failed flip halts the sequence safely (tracking restored, guider resumed) rather than imaging on a mis-aimed scope. The meridian-flip trigger is now wired into the sequencer, so a saved sequence with a flip trigger actually executes it. This is the building block auto-framing and the post-meridian-flip re-centering use. Built on the ¬ß18.I plate-solver; needs a connected mount + a configured solver to run.
- **¬ß18.I ‚Äî Plate-solve a captured frame over REST.** A new endpoint (`POST /api/v1/platesolve/frames/{id}/solve`) takes a frame from your catalog, runs it through the plate-solver, and returns the frame's true sky coordinates, image rotation, and pixel scale. - **¬ß18.I ‚Äî Plate-solving service (image ‚Üí astrometric solution).** The daemon can now plate-solve a captured frame: given an image (and optionally the mount's approximate position to seed a fast non-blind solve), it runs the solver configured in your profile and returns the frame's true sky coordinates, orientation, and pixel scale. The solver backends (ASTAP and others) were already ported but weren't reachable from anywhere ‚Äî this wires them up. - **¬ß58 ‚Äî Meridian flip trigger (decision logic).** A sequence can again decide *when* a German-equatorial mount needs to flip across the meridian: the re-ported trigger watches the mount's reported time-to-flip and, accounting for the configured pause-after-meridian / max-wait windows, the next exposure's length, and (when available) the reported side-of-pier, decides whether a flip is due ‚Äî skipping cleanly when the mount is disconnected, parked, at home, or not tracking. - **¬ß59 ‚Äî Autofocus curve fitting (Classic AF: parabolic, hyperbolic, trendlines).** The autofocus math foundation: given a set of `(focuser position, HFR)` samples from a focus sweep, the daemon fits the focus curve and reports the predicted best-focus position, the HFR it expects there, and a fit-quality R¬≤ score ‚Äî plus honest flags for "this isn't actually a focus minimum" (a flat or inverted curve) and "the minimum is outside the range you sampled, re-centre and re-run." All three classic methods are available: a weighted parabola (samples with more detected stars count more), a hyperbola (the classic focus-V model), and trendlines (a straight line through each arm of the V, intersected ‚Äî robust when the curve bottom is rounded or flat). A best-of selector keeps the parabola unless its fit quality drops below threshold, then falls back to the hyperbola. Pure managed math, identical across platforms, built on the ¬ß2105 HFR measurement. The live focuser sweep that feeds it is the next follow-up.
- **¬ß2105 ‚Äî Star detection + HFR on captured frames.** The image pipeline can now measure a frame's stars headlessly: a from-scratch detector estimates the sky background (median + MAD), thresholds at a sensitivity-driven level, groups bright pixels into stars, and reports the count plus the average Half-Flux-Radius (HFR) ‚Äî the focus-quality metric that flows into the FITS `HFR`/`StarCount` keys. Sensitivity (Normal/High/Highest) trades faint-star recall for noise rejection, and an optional median pre-filter cleans noisy subs. Pure managed code (no native OpenCV dependency), so it runs identically across Linux/macOS/Windows. This unblocks autofocus (¬ß59), which needs per-frame HFR.
