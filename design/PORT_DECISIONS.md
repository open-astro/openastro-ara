## 2026-05-26 — Phase 10 (Linux smoke test)

### `linux-x64` publish dropped from CI

- **Decision:** The `server-build` CI job + Dockerfile target **`linux-arm64` only**. The `dotnet publish -r linux-x64 --self-contained` line from playbook §11.1 is intentionally **not** carried over.
- **Reason:** Per §13 the actual deployment target is Debian 13 / RPi 4-5 (arm64). Building an x64 self-contained artifact every CI run buys nothing — there's no x64 deployment surface, no x64 Docker image, and no developer who would run `./publish/x64/OpenAstroAra.Server` against the daemon (local dev uses `dotnet run` from the source tree). User confirmed 2026-05-26 in the closed-PR-#45 thread.
- **PORT_PLAYBOOK.md §11.1 reconciliation:** §11.1 still lists both RIDs as authoritative; a separate doc PR should strike the `linux-x64` line. Until then, this decision-log entry is the controlling reference.
- **Reversibility:** trivial — adding `linux-x64` back is a 4-line CI-yaml addition.

### Base image: `10.0-noble-chiseled-arm64v8` (not `10.0-bookworm-slim-arm64v8`)

- **Decision:** `Dockerfile` uses `mcr.microsoft.com/dotnet/runtime-deps:10.0-noble-chiseled-arm64v8` as the base, **not** the playbook §11.2 example `mcr.microsoft.com/dotnet/runtime-deps:10.0-bookworm-slim-arm64v8`.
- **Reason:** Microsoft no longer publishes Debian-based (`bookworm`, `trixie`) images for the .NET 10 line — only `noble` (Ubuntu 24.04 LTS) and `azurelinux3.0`. The playbook's `bookworm-slim-arm64v8` tag returns 404 from MCR (confirmed in CI run 26477736713 on closed PR #45 successor PR #46). `noble-chiseled` is Microsoft's current minimal/distroless variant for self-contained .NET apps — ~12MB, no shell, no package manager, just the .NET runtime deps the bundled app needs.
- **§13 RPi compatibility:** the §13 RPi target is Debian 13 (Trixie), but Docker image base OS is independent of host OS. A chiseled Ubuntu container runs fine on a Debian host. The RPi's `apt`, `systemd`, etc. are unaffected by what's inside the daemon's container.
- **`USER 1000` still works** even though chiseled has no `/etc/passwd`: the kernel uses the numeric UID directly. Matches typical RPi `pi` user UID 1000.
- **PORT_PLAYBOOK.md §11.2 reconciliation:** §11.2 still lists the `bookworm-slim-arm64v8` tag as authoritative; the same separate doc PR that strikes `linux-x64` from §11.1 should also update the Dockerfile example here.

## 2026-06-02 — Workflow simplification + `master` branch protection

### Retired the `port/ara` integration branch — direct-to-master model
- **Decision:** every PR now branches from `master` and merges **directly back to `master`**. The `port/ara` integration branch and the periodic `port/ara → master` promotion step (decided 2026-05-23, §22.0) are retired.
- **Reason:** this is a single-developer port. The two-step model (sub-PR → `port/ara`, then a separate `port/ara → master` promotion per phase) doubled the PR count with no batching benefit, since promotions already happened every phase. Standard GitHub Flow (branch → PR → `master`) gives the same small-revertable-unit safety and keeps `master` continuously current, with half the ceremony. User direction 2026-06-02: *"making a branch for all work, submit PR and then merge into main would be the ideal way ... it's just me."*
- **Branch naming:** the old flat-name workaround (`phase-12h`) was forced only because `port/ara` existing as a branch made `port/ara/...` an illegal Git ref. With `port/ara` gone, that constraint is lifted — new convention is `phase/<N>[-<letter>]-<short-name>` (slash namespace + hyphens, e.g. `phase/38k-13-focuser-mediator`). User chose this 2026-06-02.
- **Default branch stays `master`** (not renamed to `main`) per user 2026-06-02.
- **Encoded in:** PORT_PLAYBOOK.md §0 rule 3 + rule 9, §1, §3 sub-PR rhythm, §19.1 branch allowlist + merge method, §22 (retitled "Direct-to-master merge model + Phase 15 final pass"); COMMIT-PR-RULES.md header note + branch-tree diagram + per-phase rhythm + review-loop step 5; `.github/workflows/ci.yml` push trigger (dropped `port/ara`, now `master` only).

### §36 Sky Atlas embed — `webview_cef` (Chromium/CEF) for the cross-desktop Aladin Lite WebView
- **Decision:** the §36 Sky Atlas embeds Aladin Lite via the **`webview_cef`** package — a Chromium Embedded Framework browser rendered to a Flutter texture — so the atlas composites **in-tab** on macOS, Windows, and Linux through one code path. Chosen by the user 2026-06-14 after the cross-desktop webview options were surfaced (per the standing v0.1.0 goal: "surface the cross-desktop webview decision before the client embed").
- **Why not the alternatives:** no single Flutter package gives a *true in-widget* embed on all three desktop targets. `flutter_inappwebview` covers macOS + Windows but **not Linux**; `desktop_webview_window` covers all three but opens a **separate OS window** (breaks the §25.5.4 in-tab layout — search/mode toggle can't overlay the atlas); native platform-views per OS (WKWebView/WebView2/WebKitGTK) embed in-tab but are three native impls with immature Windows/Linux Flutter platform-view support. The user explicitly wanted the atlas *inside the application* uniformly, which `webview_cef`'s texture-based CEF embed delivers with the least custom native code.
- **Accepted trade-offs:** (1) **bundle size** — CEF ships Chromium (~100–150 MB), which pushes against §36.1's "slim ~50 MB installer" premise *for the desktop client* (the Pi/server installer is unaffected; the atlas is a WILMA/client concern). (2) **API maturity** — `webview_cef` 0.2.2 is "under heavy development, APIs not stable yet"; pinned to `^0.2.2`. (3) per-platform setup: a `windows/runner/main.cpp` tweak (CEF multi-process + IME) lands with the Windows build slice; macOS uses CocoaPods (no SPM yet — non-fatal `flutter pub get` warning).
- **CI safety:** the native CEF build runs only on `flutter build`/`run`, not on the client CI job (`flutter analyze` + `flutter test`), which run on the Dart host — verified `pub get` + analyze + 465 tests all pass with the dep added. [[AladinView]] guards init failure (unsupported host / headless test env → an "atlas unavailable" panel) so it never crashes the tab.
- **Encoded in:** `client/.../widgets/sky_atlas/aladin_view.dart` (the embed), `pubspec.yaml` (`webview_cef` git ref). Offline-bundling the Aladin Lite JS (per §36.1, ~5 MB) + Tonight's-Sky projection wiring are follow-up §36 slices.
- **Follow-up — webview_cef was forked (2026-06-14).** Building the embed surfaced that upstream `webview_cef` (pub `0.2.2` *and* its git `main`) no longer compiles against Flutter 3.44+: Flutter added `bool TextInputClient.onFocusReceived()`, which webview_cef's `WebeViewTextInput` mixin predates, so `WebViewState` is missing an implementation. CI didn't catch it (the client job is analyze+test, which never compiled the WebView widget — no test imported it), but any real `flutter build`/on-device run would fail. The package is ~18 months stale (no maintained fix upstream or on `main`). Decision (user 2026-06-14): **fork it to [open-astro/webview_cef](https://github.com/open-astro/webview_cef)** — the same org-fork model as ASTAP + openastro-phd2 — carrying a one-line `onFocusReceived` no-op shim, and **own our line** (don't track the dead upstream; re-shim on future Flutter bumps, cherry-pick upstream only if it revives). The app pins the fork by commit SHA for reproducibility. Alternatives weighed + rejected: jnschulze/flutter-webview-windows is best-in-class but **Windows-only** (WebView2), so it can't be a cross-desktop base; the inappwebview+flutter_linux_webview hybrid is maintained but two engines (reverses the single-codepath rationale). webview_cef remains the only single package that embeds in-tab on all three desktops. **Verified:** the shim makes the client compile + all client tests pass; the full native `flutter build macos` (CEF download + tile render) remains the on-device gate. **Second fork fix (2026-06-15, SHA `2cbb28bc`):** the first on-device macOS run surfaced a runtime crash the analyze+test CI can't catch — `WebviewManager.createWebView()` threw `type 'Null' is not a subtype of type 'InjectUserScripts'` whenever a caller omitted `injectUserScripts` (which `AladinView` does), because `_injectUserScripts`/`_tempInjectUserScripts` were declared `Map<int, InjectUserScripts?>` but initialized with non-nullable `<int, InjectUserScripts>{}` literals (the literal's reified type wins, so inserting `null` throws). `AladinView`'s guard caught it and showed the misleading "Sky atlas unavailable / Chromium runtime required" fallback. Fixed in the fork by initializing both maps with the matching nullable literal `<int, InjectUserScripts?>{}`; re-pinned `pubspec.yaml` to the new SHA. This is the "own our line" model working as intended — the on-device run is the only gate that exercises the native WebView path. Provisioning (`scripts/provision-cef-macos.sh`) resolves the active checkout from `pubspec.lock`'s `resolved-ref`, so a re-pin + re-provision is one `flutter pub get` + one script run.

### §36/§25.5 — merge Sky Atlas + Framing Assistant into one **Planning** tab (2026-06-15)
- **Decision (user 2026-06-15):** the separate **Framing Assistant** and **Sky Atlas** tabs collapse into a single **Planning** tab. One Aladin Lite surface where the user **finds** a target (browse / search / Tonight's Sky) and **sets it up for a sequence** (framing → "Build Sequence") without a tab switch or the old "Set as Target in Framing Assistant" handoff CTA.
- **Why (and why it's especially right for ARA):** NINA keeps the two apart because they were *different technologies* — a list/catalog browser vs a custom sky-chart framer. ARA designed **both on the same Aladin Lite engine**, so keeping them separate would mean **two Chromium/CEF webview instances** (~100–150 MB RAM *each*, doubled) and **duplicated offline plumbing** for the same engine. Caught at the right moment: `sky_atlas_tab` has the working Aladin embed; `framing_tab` is still a stub (rotation/mosaic sliders, no sky chart, no FOV overlay — "Phase 12c.2 lights up…"), so this is a consolidation, not a teardown — we build the FOV-overlay-on-Aladin **once**, before 12c.2 ever stands up a second webview.
- **The Planning surface:**
  - **Explore** (default) — Aladin browse + universal search (Simbad online / bundled name index offline).
  - **Tonight's Sky** — location-aware "best objects tonight" from the profile's site lat/long + date, ranked by altitude/visibility; keeps the planetarium view (ARA *is* the planetarium — no external Cartes du Ciel/Stellarium).
  - **Frame (toggle)** — overlays the **profile-derived FOV**: camera sensor (W×H px) + pixel size + telescope effective focal length → `pixelScale = 206.265 × pixelSize_µm ÷ focalLength_mm`, FOV rectangle = `sensor_px × pixelScale`, plus rotation + mosaic grid (§47). Output is **"Add to Sequence" / "Build Mosaic Sequence."** No manual FOV entry — it shows how the object will actually look through *this* user's rig.
- **Profile-cached camera geometry (key enabler):** Planning is done **away from the rig** (couch, camera disconnected, maybe offline), so the FOV box can't depend on a live camera. On the **first successful camera connect** for a profile, ARA persists the sensor geometry (sensor W/H px, pixel size, and effective focal length incl. reducer/barlow) into the profile; thereafter framing "just works" without the camera connected, online or off. Manual override stays available in Settings → Camera/Telescope. (Data model already carries the inputs: `CameraSettings.PixelSize`, `TelescopeSettings.FocalLength`, `EquipmentDtos.SensorWidth/Height`.)
  - **Cache invalidation (must not go stale):** a swapped camera or changed reducer/barlow on an existing profile would otherwise silently produce a wrong FOV + mosaic grid. Re-cache on three triggers: (a) any camera connect that reports sensor dimensions **differing** from the cached values (auto-update + one-time notice); (b) the existing **§30.7 equipment-change check** — its invalidation matrix already covers "telescope/focal length → framing" and "camera → dither magnitude", so the camera-geometry cache joins that recompute path; (c) a manual **"Refresh from connected camera"** affordance next to the FOV readout when a camera is connected. The implementation PR specifies the exact UX.
- **Naming / discoverability:** the tab is **"Planning"** (not "Framing" or "Sky Atlas"). §61 smart-search maps "framing", "sky atlas", "FOV", "mosaic", "tonight" to it so NINA users and search both land here.
- **Scope:** client-side UX consolidation — server (mosaic endpoints, build-sequence, Data Manager) is unaffected. `framing_tab`/`framing_state` fold into the Planning surface on the existing Aladin embed. The detailed §36 (Sky Atlas) + §47 (mosaic) + §25.5 (layout) + §61 (registry) prose is reconciled in the implementation PR; the tab list + §-UI walkthrough are updated here for consistency.

### §36 Planning — lean on the Stellarium Web engine's NATIVE capabilities; don't rebuild the wheel (2026-06-26)
- **Principle (user 2026-06-26):** "Use the internal engine for everything it offers and scrap the rest. Nonsense rebuilding the wheel… The goal is to build an enterprise application but to also lean on the software we are implementing from other sources. Some of these tools already do what we want — we are just refining them to be better and more perfected." This generalises beyond the planetarium: for every vendored/integrated engine (Stellarium Web, Aladin, openastro-phd2/guider, ASTAP, NINA), map and use its **native** capability before writing custom code; only fill the genuine gap, and keep it minimal. Don't overthink it.
- **What the Stellarium Web engine already does natively (so we DON'T reimplement):**
  - **Deep-sky display** — `core.dsos` renders the full bundled DSO catalog (NGC / IC / Messier / Caldwell; designations parsed in `src/designation.c`, which knows `M`/`C`/NGC/IC) as a tiled HiPS survey with **magnitude-scaled hint circles + names**, GPU-batched — SkySafari-quality, thousands of objects, no lock. Controlled by `core.dsos.visible`, `hints_visible`, `hints_mag_offset`.
  - **Search by designation** — `core_search` / `stel.getObj("NGC 7000" | "M31" | "Vega" | …)` resolves objects by any catalog designation; already wired to the Planning search bar.
- **What it does NOT do natively:** filter the *display* down to a single catalog/type (it exposes only visible/hints_visible/hints_mag_offset — no per-catalog show-filter).
- **Decision:** SCRAP the custom catalog system that rebuilt the wheel — the daemon's `SkyCatalogService` + `GET /api/v1/catalogs[/{id}]` endpoints and the client's per-object **GeoJSON catalog overlays** (`toggleCatalog`/`catData`/`catalogGeojson` in `assets/stellarium/index.html`). Those locked the renderer at a few hundred objects while the engine renders the whole catalog natively. The Planning "Catalogs/Display" controls should drive the engine's **native** knobs (deep-sky on/off, hint density via `hints_mag_offset`) and rely on native search — not custom overlays. (A regression had set `core.dsos.visible=false` at startup, hiding the DSOs that "used to show all the NGCs"; restored to `true`.)
- **Enabler shipped same day:** real macOS GPU rendering for the engine — `webview_cef` fork now uses shared-texture OSR (`OnAcceleratedPaint` → IOSurface → Flutter texture) instead of `--disable-gpu`/SwiftShader software WebGL, plus a fix to the IOSurface use-count abort that path exposed. The native DSO layer renders smoothly on the GPU.
- **Encoded in:** `assets/stellarium/index.html` (native `core.dsos` on; overlay machinery to be removed), `packages/webview_cef` (GPU OSR, branch `feat-macos-gpu-osr`). See memory `feedback_lean_on_vendored_engines`, `project_catalog_overlay_epic`.

### §36 — rebuild the Stellarium WASM to make EVERY star clickable (2026-06-27)
- **Problem:** the engine only made *labelled* objects selectable. `core_get_obj_at` (core.c) picks from rendered **labels** + registered **areas**; DSOs register a pickable area unconditionally (so NGC/IC always clicked), but stars only got an area when `luminance>0.5 && size>1` (a code comment literally said *"This makes very faint stars not selectable"*). No JS setting changes it — proven: Vega (mag 0) clicked, 51 Peg (mag 5.4, *named*) didn't, even with `hints_mag_offset=6`. The engine exposes no JS pick/unproject either (`window.Module` is closure-scoped; only `_core_lookat`/`_core_on_mouse`/… are exported), so a JS-side nearest-star picker isn't possible.
- **Decision (user 2026-06-27, "rebuild the engine"):** recompile the vendored WASM so the stars module registers a pickable area for **every rendered star** (`stars.c` — drop the brightness gate), with a **per-frame budget** (`STARS_MAX_AREAS = 4000`, brightest-first since tiles are vmag-sorted) so deep, dense fields don't register tens of thousands of areas and stall the renderer. Also relaxed `stars_add_data_source` to accept a star survey whose `properties` omits the `type` key. **Verified on-device:** click any star → panel with its designation; no lock.
- **Build:** source lives in **`~/Documents/GitHub/stellarium-web-engine-build`** (not vendored in this repo — only the built `.js`/`.wasm` are, in `assets/stellarium/`). ### §36 — planetarium webview: CEF OSR stalls on macOS; pivot toward native WKWebView (2026-06-27)
- **Finding:** the `webview_cef` GPU offscreen-render path (OSR → IOSurface → Flutter texture, enabled this session) **freezes on macOS** under sustained render — lock-up opening the framing overlay in a dense field, and an **idle freeze** that kills even zoom after minutes. Default builds stay on CEF (`kUseWebViewAll` in `stellarium_view.dart`). The swap is low-risk here because the page is self-driven and **all Flutter↔page comms go over the loopback asset server** (`/aracmd` poll + `/araevent` post), not a webview JS bridge — so any webview that loads a URL works.
- **Why `webview_all`:** the only candidate that natively covers all three desktop OSes *and* embeds in-tree. `flutter_inappwebview` (very mature) was rejected for **no Linux**; `atomic_webview` for opening a **separate window**. Trade-offs of the native-platform-view approach: Flutter widgets can't reliably float *over* it (our planetarium UI is all in-page HTML, so fine), and platform views have clipping/transform quirks.
- **Direction:** lean toward retiring `webview_cef` and going **native-engine-per-platform** via `webview_all`. Gated on a Linux WebKitGTK WebGL2 spike (very likely fine on x86-64 desktop Mesa). See PORT_TODO. ## 2026-07-01 — Licensing: WILMA client goes AGPL-3.0-or-later; server stays MPL-2.0

### Client relicensed to deter proprietary hardware-vendor absorption (user, 2026-07-01)

- **Decision:** the WILMA desktop/mobile client (`client/openastroara_client/`) is licensed **AGPL-3.0-or-later** going forward (per-directory `LICENSE` in the client root). The server/daemon and everything else in the repo stay **MPL-2.0** — required anyway for the NINA-derived files, whose MPL lineage is unchanged.
- **Why (user):** MPL-2.0 is file-level copyleft — a closed-hardware vendor (the user named ZWO/ASIAIR-style products as the concern) could absorb the client and keep all additions proprietary in new files. AGPL's network-interaction clause (§13) means any device that ships *or serves* a derived client must publish the complete combined source — an effective commercial deterrent. The client is wholly original work (Dart/Flutter, written for this project; no NINA-derived files), so the org is free to choose its license.
- **Boundary is legally clean:** client and daemon communicate over REST/WS as separate programs — no linking, so AGPL (client) and MPL-2.0 (daemon) never contaminate each other. The client already bundles the AGPL Stellarium Web Engine, so its own AGPL is coherent with its contents. Historical revisions already published under MPL-2.0 remain MPL at those revisions (pre-release snapshots).
- **Repo layout unchanged:** the client stays in the monorepo (it is generated/developed against the daemon's OpenAPI surface; splitting repos was considered and rejected). Licensing is per-directory, which the README/NOTICE now document.
- **Related (separate repo):** the maintainer asked whether `open-astro/AlpacaBridge` (currently SSPL) should also move to AGPL-3.0. Recommended **yes**: SSPL is not OSI-approved and Debian/Fedora treat it as non-free (blocks distro/apt inclusion — ARA's own .deb `Recommends: alpacabridge`), while its extra-strength clause only targets SaaS resale, not the hardware-vendor threat AGPL §13 already covers. ## 2026-07-07 — §35 unsafe-reaction engine semantics

### Reaction semantics chosen for v0.0.1 (AI, per the standing design mandate)

- **`pause_and_park` does NOT abort the in-flight exposure.** The §38 pause gate suspends at the next instruction boundary (NINA semantics), while the mount parks immediately — so during a long sub the frame is sacrificed (mount moves under an open shutter) but the gear is protected NOW and the run suspends cleanly + resumably. Aborting the exposure instead would make the running `TakeExposure` instruction fail and could drive the run to `Failed` (not `Paused`) — protecting a frame that is already lost at the cost of resumability. Users who want the harsher semantics pick `abort_and_park`.
- **Auto-resume releases the run where it stood; it does not re-slew.** After unpark the mount points at/near park position; the engine waits (bounded 90 s) for the runtime to read back unparked, restores tracking, releases the runs, and the notification explicitly tells the user to verify pointing unless the sequence re-centers. Re-slew-before-release needs per-run target coordinates surfaced to the engine — recorded as a v0.1.0 refinement in PORT_TODO. Honest-but-imperfect beats silently shooting the park position without warning the user.
- **`PausedAwaitingUser` is never auto-resumed and automated safety actions are never §58.12 user activity.** New `ISequencerService.PauseActiveRunsAsync`/`ResumeRunsAsync` exist precisely so the engine doesn't ride the user-facing `PauseAsync`/`ResumeAsync` (whose `NotifyUserActivity` would cancel an unattended-shutdown countdown as if the user came back). The bulk pause also skips already-paused runs so the engine can never "adopt" (and later auto-resume) a pause the user or a flip failure owns.
- **A disconnected monitor is unknown, not safe.** Losing the SafetyMonitor cancels a pending auto-resume (no evidence conditions are safe) but triggers no new action (no evidence they're unsafe either); a monitor that CONNECTS reporting unsafe triggers immediately (unknown→unsafe counts as a transition).
- **Unknown `on_unsafe` tokens fall back to `pause_and_park`** (fail toward protecting the gear; logged so a typo'd profile is visible rather than silently inert).
